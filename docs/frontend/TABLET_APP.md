# Purpose

This document specifies QAYD's tablet experience — iPad (iPadOS Safari and the installed PWA) and Android tablets (Chrome and the installed PWA) — as a deliberate, first-class layout tier of the platform's one frontend codebase, not a second application. A note on scope, stated plainly because earlier planning material for this document series described tablet coverage as "Flutter native + responsive web": that framing predates, and conflicts with, the platform's fixed frontend stack. `docs/frontend/FRONTEND_ARCHITECTURE.md` and `docs/foundation/TECH_STACK.md` are explicit and singular — Next.js 15, React 19, TypeScript strict, Tailwind CSS, shadcn/ui on Radix primitives — and nowhere in the platform's foundation, API, or frontend documentation does a Flutter (or any other native-mobile) codebase exist alongside it. QAYD ships **exactly one frontend**. What a tablet user opens is that same Next.js application, installable to the home screen as a PWA (the "native shell / PWA" entry already named in `docs/frontend/RESPONSIVE_DESIGN.md`'s gesture catalog for edge-swipe-back), running inside the same App Router, the same TanStack Query cache, the same Laravel Reverb realtime layer, and the same design tokens every desktop and mobile screen in this documentation series already uses. This document does not introduce a parallel rendering pipeline, a parallel component library, or a parallel API contract — it specifies how the *existing* responsive system in `docs/frontend/RESPONSIVE_DESIGN.md`, `docs/frontend/LAYOUT_SYSTEM.md`, and `docs/frontend/NAVIGATION_SYSTEM.md` behaves, and where it should be deliberately extended, at tablet width. If a genuinely separate native tablet shell (Capacitor, or a platform-native Swift/Kotlin client) is ever commissioned, it supersedes this document and needs its own specification; until then, this is the authoritative statement of what "the tablet app" means for QAYD.

Tablet matters to QAYD for reasons specific to a finance product, not generic "responsive design completeness." The platform's own stated tablet use cases — a Finance Manager reviewing a Trial Balance or a reconciliation from a client site, a Warehouse Employee running a stock count on a shop floor, a CFO annotating a vendor bill before a Board pack goes out — are named directly in `docs/frontend/LAYOUT_SYSTEM.md → Touch targets` ("Tablets are a real QAYD use case") and in `docs/frontend/RESPONSIVE_DESIGN.md`'s own founding framing (data entry on a handheld device is "the hardest version of every input problem QAYD has"). None of those are casual-browsing scenarios; they are the same work a desktop user does, on a device that happens to be handheld, propped on a stand, or paired with a keyboard case. A tablet tier that merely stretches the mobile layout wastes that canvas on whitespace with no added structure; a tablet tier that force-fits the full desktop shell (a 272px labeled sidebar, a four-pane reconciliation board) either does not fit the viewport or crowds out the touch-comfortable spacing a handheld device still needs. This document's job is to specify the middle path QAYD actually ships: master-detail and two-pane patterns that activate specifically at this tier, a navigation chrome calibrated to the tier's own constraints rather than borrowed wholesale from either neighbor, and journeys — reviewing a document beside its approval card, reconciling a statement two-pane, working a dashboard's bento grid, entering a journal line at a touch-safe row height — that are demonstrably better on a tablet's canvas than on a phone's, without requiring the mouse precision or persistent chrome a laptop assumes.

# Platform Strategy

**Tablet is a CSS-viewport tier, not a device category, and it is not a single tier.** This is the load-bearing architectural fact the rest of this document builds on, and it is already established, precisely, across three sibling documents that this one only has to apply rather than re-derive. `docs/frontend/RESPONSIVE_DESIGN.md`'s tier-mapping table names five semantic tiers — Mobile, Tablet, Laptop, Desktop, Ultra Wide — mapped onto Tailwind's `sm`/`md`/`lg`/`xl`/`2xl`/`3xl` breakpoints, and it is explicit that these tiers track **CSS pixel width, never a named device**: "a '1366×768 laptop' and a '1024px iPad' can render an identical layout, and QAYD wants that." The consequence for a physical tablet is that it does not live in one tier — it moves between two, deterministically, based on orientation and model:

| Physical device | Orientation | CSS width | Platform tier | What renders |
|---|---|---|---|---|
| iPad mini (6th gen) | Portrait | 744px | **Tablet** (`md`, 768 boundary not yet cleared — see Edge Cases) | Off-canvas nav Sheet, single-column-to-two-pane list/detail |
| iPad Air / iPad 11" | Portrait | 820–834px | **Tablet** (`md`) | Off-canvas nav Sheet, master-detail split viable |
| iPad Pro 11" | Landscape | 1194px | **Laptop** (`lg`) | Persistent collapsed icon rail, full two/three-pane layouts |
| iPad Pro 12.9" | Portrait | 1024px | **Laptop** (`lg`) — sits exactly on the boundary | Persistent collapsed icon rail even in portrait |
| iPad Pro 12.9" | Landscape | 1366px | **Laptop**/**Desktop** boundary (`lg`–`xl`) | Full desktop-equivalent chrome |
| Android tablet (10–11") | Portrait | 800px | **Tablet** (`md`) | Off-canvas nav Sheet |
| Android tablet (10–11") | Landscape | 1280px | **Desktop** (`xl`) | Full desktop chrome, four-column KPI strip |

This document's scope is therefore both of the tiers a physical tablet actually renders in — **Tablet** (`md`, 768–1023px) and the tablet-relevant portion of **Laptop** (`lg`, 1024–1279px) — plus the transition between them a single physical device crosses every time its owner rotates it. Where the two tiers diverge meaningfully (a persistent nav rail at `lg` versus an off-canvas Sheet at `md`; a three-pane reconciliation board at `lg` versus a tabbed two-surface fallback at `md`), this document specifies both states and the rotation between them explicitly, rather than picking one and calling the other "close enough."

**Tablet is where several of the platform's own documents already agree a genuine second layout tier begins**, and this document treats that agreement as its foundation rather than inventing new behavior from nothing:

- `docs/frontend/RESPONSIVE_DESIGN.md → Pattern 2` places the sidebar's Sheet-versus-inline-rail decision at `lg`, not `md`, on the record of the platform's own Kuwait pilot-cohort testing: "a 72px icon-only rail on a 768–1023px tablet left too little room for the two-pane list/detail layouts that tier is otherwise wide enough for." Tablet portrait deliberately trades a persistent rail for the width a master-detail split needs — this is not an oversight, it is the documented reason tablet gets its own chrome rather than inheriting the laptop's.
- `docs/frontend/REPORTS.md`'s Report Builder gives tablet its own named middle layout — "between `md` and `xl` it renders two panes (Fields collapses into a `Popover`...) rather than jumping straight to the mobile wizard, since a tablet in landscape has room for two panes comfortably" — the exact "reduce by one pane, don't collapse to a wizard" pattern this document generalizes in `# Layout` below.
- `docs/frontend/APPROVAL_CENTER.md`'s own Responsive Behavior table names Tablet (`md`–`lg`) as a distinct middle column between Desktop/Ultra Wide and Mobile, with its own specific trade-offs (Table view hidden, Card view only; header controls collapse to an overflow menu rather than a `Sheet`).
- `docs/frontend/DASHBOARD.md`'s Responsive Behavior table gives tablet a two-tile KPI Strip at `md` and the full 8/4 chart-plus-AI-rail split only once `lg` is reached — and `docs/ai/AI_COMMAND_CENTER.md → Layout` states, of its own denser grid, that "grid concepts do not apply below the tablet breakpoint at all" — i.e., the Command Center's spatial bento grid, collapsed to a single swipeable column on mobile, resumes being a grid starting exactly at tablet width, not at laptop width.

**The strategic commitment this document makes, on top of that foundation:** tablet gets its own named state in every screen this platform documents that has more than one meaningful pane, region, or density — never a silent inheritance of mobile's single column, and never a forced fit of desktop's fullest layout. Where an existing screen document's responsive table is silent on tablet specifically (folding it into "same as mobile" or "same as desktop" by omission), that silence is treated as a gap this document closes, not as evidence tablet needs nothing of its own.

# Layout

## Master-detail and the container-width split rule

QAYD's master-detail pattern — a list pane and a detail pane, visible together — is the single most consequential tablet-specific layout decision this document makes, because none of the sibling screen documents needed to solve it precisely: Desktop has the width to spare and uses `docs/frontend/LAYOUT_SYSTEM.md`'s Detail Page Template's 8/12 Main Column plus 4/12 Summary Rail; Mobile has no width to spare and pushes to a full-screen Detail route. Tablet sits in a zone where the answer depends on *exactly how much width is actually available to the Content region* — which, per `docs/frontend/LAYOUT_SYSTEM.md`'s own App Shell Layout, is not simply the viewport width; it is the viewport minus whatever the Sidebar and AI Rail are currently consuming, and both of those already vary by tier and by user choice.

The rule this document specifies: master-detail split-readiness is decided by a **container query on the Content region itself**, not a viewport breakpoint, using the exact `@container` mechanism `docs/frontend/LAYOUT_SYSTEM.md → Container queries for widget-level responsiveness` already establishes for `KpiTile`. A list+detail screen (Journal Entries, Invoices, the Approval Center's own detail route, Customers) wraps its Content region in `@container` and switches from a single, full-width list (row tap pushes a full-screen Detail route, exactly as Mobile does) to a persistent split once the container clears **640px** of usable width — a threshold chosen to guarantee the list pane (a fixed 320px, per below) and the detail pane's own minimum legible width (roughly 320px for a summary card) both fit without either degrading:

```tsx
// components/shared/master-detail/master-detail-view.tsx
'use client';

export function MasterDetailView<TRow>({ rows, selectedId, onSelect, renderRow, renderDetail }: MasterDetailViewProps<TRow>) {
  return (
    <div className="@container flex h-full min-w-0">
      {/* Below the 640px container threshold: list only, full width. A row's onClick pushes
          router.push(detailHref) — the identical full-screen Detail route Mobile uses, never a
          second, tablet-only detail implementation. */}
      <div className={cn('min-w-0 flex-1 overflow-y-auto', selectedId && '@[640px]:max-w-80 @[640px]:flex-none @[640px]:border-e')}>
        {rows.map((row) => renderRow(row, { active: row.id === selectedId }))}
      </div>
      {/* Detail pane mounts only once the container clears 640px AND a row is selected —
          otherwise the list keeps the full container and a tap navigates instead of splitting. */}
      <div className="hidden min-w-0 flex-1 overflow-y-auto @[640px]:block" hidden={!selectedId}>
        {selectedId ? renderDetail(selectedId) : null}
      </div>
    </div>
  );
}
```

This is deliberately a **content-region** container query, not a `md:`/`lg:` viewport variant, because the same 834px-wide iPad renders a genuinely different amount of Content width depending on whether the AI Rail is docked open (`docs/frontend/LAYOUT_SYSTEM.md`'s `360px` region, which on tablet only ever renders as a `Sheet` overlay per its own region table — so it never actually steals persistent width from Content on this tier, but the container-query approach means the component does not need to know that fact itself). A viewport-keyed `md:flex` would need a second, hand-maintained exception the moment any future shell region changes width; the container query is correct by construction. The list pane's own width, once split, is a fixed **320px** (not a 12-column fraction) — a deliberate choice matching `docs/frontend/NAVIGATION_SYSTEM.md`'s own Sidebar-adjacent sizing logic: a list of document numbers, statuses, and dates does not benefit from stretching proportionally the way a detail pane's prose and tables do, and a fixed list-pane width keeps the split's proportions stable as the window itself resizes under Stage Manager (`# Navigation & Multitasking`).

## Multi-column at tablet width

Outside master-detail, tablet's extra width is spent on **one additional column, never a proportional stretch of desktop's full column count**, matching `docs/frontend/DASHBOARD.md`'s own precedent precisely: the KPI Strip goes from Mobile's one-tile-per-carousel-slide to **two tiles per row at `md`**, and only reaches its full three-to-six-tile row at `lg`/`xl`. Two-column form grids activate at the same tier per `docs/frontend/RESPONSIVE_DESIGN.md → Forms On Mobile`'s `md:grid-cols-2 lg:grid-cols-3` rule — a tablet-portrait Journal Entry header (Date, Period, Reference, Memo) reads as two fields per row rather than the single stacked column Mobile needs or the three-per-row density `lg` affords. `docs/accounting`-adjacent aging summaries (AR/AP side by side) hold to the platform's existing rule that this specific pairing waits for `lg` — two half-width bars at `md` would each be narrower than either bar's own minimum legible width for a four-bucket breakdown, so tablet portrait keeps them stacked full-width and gains the side-by-side pairing only in landscape, exactly as `docs/frontend/DASHBOARD.md`'s Responsive Behavior table already specifies.

## Sidebar behavior across the tablet range

Restated precisely because it is the single most-tested assumption in this document: **there is no persistent sidebar rail anywhere in the Tablet tier (`md`, 768–1023px).** The primary nav lives behind the Topbar's hamburger trigger, opening the full nav tree as a `Sheet` sliding from the logical start edge — identical in mechanism to Mobile, per `docs/frontend/RESPONSIVE_DESIGN.md → Pattern 2` and `docs/frontend/LAYOUT_SYSTEM.md`'s own per-breakpoint table ("`md` … Sidebar becomes an overlay drawer, not inline"). The instant a tablet crosses into the Laptop tier (`lg`, 1024px+) — by rotating to landscape, or simply being a 12.9"-class device whose portrait width already clears 1024px — the Sidebar becomes a persistent, **collapsed** 72px icon rail (never the 272px labeled rail; that additionally requires `xl`, per `docs/frontend/NAVIGATION_SYSTEM.md`'s region contract), and the `Cmd/Ctrl+B` toggle to expand it becomes available for the first time. A tablet user who habitually works landscape with an external keyboard therefore experiences the identical rail-collapse/expand affordance a laptop user does; a tablet user working portrait, handheld, never sees a rail steal width from the master-detail split it would otherwise support.

## Bottom navigation does not extend to tablet

`docs/frontend/RESPONSIVE_DESIGN.md → Pattern 3` scopes `BottomNav` to `md:hidden` — it disappears at the same 768px boundary the Sidebar-as-Sheet begins, not later. Tablet therefore sits in a real navigational gap between Mobile's thumb-reachable bottom bar and Laptop's persistent rail: primary navigation on a tablet in portrait is reached exclusively through the Topbar's hamburger `Sheet` trigger and the `⌘K` Command Palette (both available identically to a Bluetooth keyboard's actual `Cmd+K` chord, per `# Navigation & Multitasking`). This is a deliberate consequence of the platform's own rule, not an omission this document patches over with a bottom bar of its own — a five-item bottom bar competing with a two-pane master-detail split for vertical space would undercut the exact width/height budget the split needs, and the Command Palette already gives tablet a faster, keyboard-and-touch-both-friendly path to any destination than a five-icon bar would.

## List+detail for ledgers and approvals

Every list+detail screen this platform documents — Journal Entries, Invoices, Bills, Customers/Vendors, and the Approval Center's own `[approvalId]` detail route — adopts the `MasterDetailView` pattern above at tablet width, with one screen-specific adjustment each:

- **Journal Entries / Invoices / Bills.** The list pane shows `docs/frontend/LAYOUT_SYSTEM.md`'s List Page Template columns filtered to `priority: 1–2` (document number, counterparty, amount, status — identical to the fields `docs/frontend/RESPONSIVE_DESIGN.md`'s card-transformation rule already selects for Mobile's stacked cards), and the detail pane renders the full Detail Page Template's Main Column, with its own 4/12 Summary Rail collapsed into a `Sheet` triggered from the detail pane's own header rather than a third simultaneous column — three real columns (list, detail, summary rail) do not fit tablet width even at the widest Tablet-tier iPad, and forcing them in would violate the same minimum-pane-width guarantee the 640px split threshold exists to protect.
- **Approval Center.** `docs/frontend/APPROVAL_CENTER.md`'s own Responsive Behavior table already specifies Card view only (Table view hidden) for `md`–`lg` — this document's master-detail split applies on top of that: tapping a card opens the detail pane beside the still-scrollable card list rather than navigating away from the queue entirely, so a Finance Manager working through a backlog on an iPad keeps the list's scroll position and remaining-count visible while reviewing each item, rather than round-tripping back to the queue after every decision.
- **Customers / Vendors.** These follow the identical `MasterDetailView`, with the list pane's fixed 320px width comfortably holding a bilingual (`name_en`/`name_ar`) name plus a status `Badge`, matching the Relaxed density row height (48px) `docs/frontend/DESIGN_LANGUAGE.md → Data Density` already sets as this data class's default.

# Navigation & Multitasking

## Split View and Slide Over

QAYD makes no attempt to detect iPadOS Split View, Slide Over, or Android's split-screen mode as a distinct condition, and this is correct by construction rather than a gap: per `docs/frontend/RESPONSIVE_DESIGN.md → Mobile-First Strategy`, layout is decided entirely by live CSS viewport width through Tailwind's media queries and the shared `useBreakpoint()` hook, never by a device-type or multitasking-state check. A QAYD window placed into a 50/50 Split View on an 11" iPad (834pt wide) receives roughly 390–410pt per app — squarely inside the **Mobile** tier — and renders the bottom-tab-bar-and-stacked-card layout exactly as it would on a phone, correctly, with no special code path required. The same window widened to a 70/30 Split View still typically lands under 640pt (still Mobile tier); only a 12.9"-class iPad's Split View, or a Slide Over promoted to full width, reliably clears into Tablet or Laptop tier. The practical guidance for anyone building a new screen against this document: **never gate a control on "is this a tablet" — gate it on the live container/viewport width, exactly as every sibling document in this series already does**, because a tablet in Split View is, for layout purposes, temporarily a phone, and the reverse is equally true of a phone-class device connected to an external monitor.

## Stage Manager and arbitrary window widths

Stage Manager (M-series iPads) and equivalent free-form windowing on Android tablets present QAYD in an arbitrarily resizable window rather than one of a fixed set of Split View halves, which makes the width genuinely continuous rather than a handful of known values. The responsive system already handles the continuum correctly by the same reasoning as Split View above. The one addition this document makes for the two screens whose value proposition depends specifically on two panes fitting side by side — Bank Reconciliation's Match Board and the Report Builder — is a soft, dismissible nudge, never a hard block: once a Stage Manager window narrows below the 640px master-detail threshold on one of those two screens specifically, a single-line, non-modal banner ("Widen this window for the full matching view") appears above the collapsed single-pane fallback (`# Priority Tablet Journeys`), dismissible per-device via `localStorage` using the identical "dismissed once per device" convention `docs/frontend/RESPONSIVE_DESIGN.md`'s sticky-table scroll-hint gradient already establishes — it never blocks the underlying single-pane workflow, which remains fully functional, per the platform's standing rule that a capability may take an extra tap at a narrower tier but is never unreachable.

## External keyboard shortcuts

Every keyboard shortcut this platform's documentation defines is input-agnostic and therefore already available on a tablet the instant a hardware keyboard — Smart Keyboard Folio, Magic Keyboard, or a Bluetooth keyboard on Android — is paired, with zero tablet-specific branching required: the global `⌘K`/`Ctrl+K` Command Palette (`docs/frontend/NAVIGATION_SYSTEM.md`), `Cmd/Ctrl+B` to toggle the Sidebar between expanded and collapsed once the tier is `lg`+ (`docs/frontend/LAYOUT_SYSTEM.md`), the non-production `Cmd/Ctrl+Shift+G` grid debug overlay, and a dense table's own arrow-key row navigation, `Enter`-to-open, and table-scoped `Cmd/Ctrl+F` filter focus (`docs/frontend/DESIGN_LANGUAGE.md → Virtualization & interaction`). A Finance Manager who docks an iPad in a Magic Keyboard case for a morning of Trial Balance review gets the identical keyboard-driven workflow a laptop user gets, because the event listeners these shortcuts bind to (`keydown` on `window`) do not distinguish a tablet's Bluetooth HID keyboard from a laptop's built-in one. This document adds one tablet-specific shortcut pairing for the master-detail split defined in `# Layout`: `Cmd/Ctrl+[` moves focus to the list pane and `Cmd/Ctrl+]` moves it to the detail pane, mirroring the logical-direction convention (`Cmd/Ctrl+[` targets the logical start pane, `Cmd/Ctrl+]` the logical end pane, so the physical keys' effect swaps correctly under `dir="rtl"` exactly as every other directional affordance in this platform already does).

## Pointer and trackpad — a capability, not a tier

`docs/frontend/RESPONSIVE_DESIGN.md → Gesture catalog` gates `Ctrl/Cmd-click, Shift-click` multi-select to "Laptop, Desktop, Ultra Wide" — a viewport-tier gate. This document refines that gate for one specific, real scenario the tier list does not cover: an iPad mini or Air paired with a Magic Keyboard or a Bluetooth mouse sits, physically, in the **Tablet** tier (`md`, 768–1023px) while carrying exactly the precise pointer that gesture was designed for. The correct condition is the pointer capability itself, not the viewport tier — and the platform already has the right primitive for this, because `docs/frontend/RESPONSIVE_DESIGN.md → Touch Targets & Gestures` independently established `@media (hover: hover) and (pointer: fine)` as the feature query that gates hover-revealed row actions. This document extends that same feature query to multi-select:

```tsx
// hooks/use-fine-pointer.ts
'use client';
import { useSyncExternalStore } from 'react';

const QUERY = '(hover: hover) and (pointer: fine)';

function subscribe(callback: () => void) {
  const mql = window.matchMedia(QUERY);
  mql.addEventListener('change', callback);
  return () => mql.removeEventListener('change', callback);
}

/** True whenever a precise pointer (trackpad, mouse) is currently available — independent of
 *  viewport tier. A Tablet-tier iPad with a Magic Keyboard's trackpad returns true here even
 *  though RESPONSIVE_DESIGN.md's own Gesture catalog table lists multi-select as Laptop+ only;
 *  this hook is what lets that gesture reach a keyboard-cased tablet correctly. */
export function useFinePointer() {
  return useSyncExternalStore(subscribe, () => window.matchMedia(QUERY).matches, () => false);
}
```

A `DataTable` row's selection handler checks `useFinePointer()` rather than `useBreakpoint()` before honoring a `metaKey`/`ctrlKey`/`shiftKey` click-to-extend-selection — and because `matchMedia` re-evaluates live, detaching the keyboard case mid-session (`# Edge Cases`) correctly and immediately falls back to the checkbox-based touch selection already present for every tier, with no stale affordance left behind. iPadOS's own "precise pointer" hover halo over a button or a table row needs no special handling beyond this: standard CSS `:hover`/`:focus-visible` styling, already gated behind the identical `hover: hover` feature query platform-wide, renders correctly the instant the OS reports a trackpad is in use, and disappears the instant a finger touches the glass instead, because iPadOS itself toggles the underlying media query in real time as the input device changes.

# Priority Tablet Journeys

## Reviewing and approving with the document side by side

This is the journey the extra tablet canvas earns its keep on most directly, and it is not fully specified anywhere else in this series — `docs/frontend/APPROVAL_CENTER.md` specifies the queue, the card, the reasoning disclosure, and the swipe gesture, but not a document-beside-decision layout, because that screen's own desktop treatment does not need one (a desktop reviewer already has the width for a `Sheet` or a second window). At Tablet-and-up width, opening an `ApprovalCard`'s detail — most commonly a vendor bill or an expense claim whose Document AI/OCR extraction is the very thing being verified — splits the detail pane itself into two: the `ApprovalCard`'s own reasoning, confidence, `ApprovalStepper`, and Approve/Reject/Request Changes/Delegate actions occupy the pane's start side, and the source attachment renders in a companion `AttachmentViewerPane` on the end side, pinch-zoom and pan enabled, reusing the same polymorphic-`attachments` viewer `docs/frontend/DOCUMENT_CENTER.md` already defines for previewing an `attachable_type`/`attachable_id` record rather than a bespoke, screen-specific PDF viewer:

```
┌──────────────────────────────┬────────────────────────────────┐
│ Vendor Bill — Gulf Office     │  [ pinch to zoom, drag to pan ] │
│ Supplies · KWD 1,140.000      │                                  │
│ ● AI · Document AI — 94%      │      [ invoice.pdf, page 1 ]     │
│ Extracted: amount, vendor,    │                                  │
│ VAT, due date all match the   │                                  │
│ submitted bill                │                                  │
│ [Reject] [Req. changes]       │                                  │
│                    [Approve]  │                                  │
└──────────────────────────────┴────────────────────────────────┘
```

A reviewer checks the AI's extracted amount against the actual printed figure without leaving the approval, downloading a file, or losing their place in the queue behind it — the master-detail list pane from `# Layout` remains visible and scrollable to the pane's own far side on a sufficiently wide tablet, giving a genuine three-region reading (queue, decision, source) that only tablet-and-up width supports comfortably. Below the 640px split threshold, the two panes stack — reasoning above, document below, both full width, scroll independently — never side by side at a width where either would drop under its own legible minimum.

## Reconciliation, two-pane

`docs/frontend/BANK_RECONCILIATION.md`'s three-part Match Board — Statement Lines pane, Match Tray, Transaction Candidates pane — is specified at `lg`+ inside a `full-bleed` container. Because tablet **landscape** routinely clears `lg` (1024px, per the device table in `# Platform Strategy`), a Finance Manager or Senior Accountant reconciling from an 11"+ iPad in landscape gets that exact three-pane board unchanged — no fallback needed, no reduced feature set, the direct payoff of the platform's CSS-tier-driven design. Tablet **portrait** (`md`, 768–1023px) does not have the width for three simultaneous panes at a legible minimum each, so this document specifies the fallback the Match tab renders below `lg`:

```
┌────────────────────────────────────────────────────────────┐
│ Difference — must reach 0.000 to close          Variance 42.750│
├────────────────────────────────────────────────────────────┤
│  Statement lines (52)    Candidates (49)                    │  ← Tabs, not side-by-side panes
├────────────────────────────────────────────────────────────┤
│ ☐ Jul 16  KNET TRANSFER   +615.000                          │
│ ● Jul 16  FT2607201234  −4,250.000  ✓ auto-matched · 100%   │
│ ○ Jul 17  MONTHLY A/C FEE   −3.500                          │
│ …                                                             │
├────────────────────────────────────────────────────────────┤
│ 1 staged  ·  [Match]  [Split]  [Clear]           ← sticky Match Tray, bottom-anchored │
└────────────────────────────────────────────────────────────┘
```

The Statement Lines and Transaction Candidates panes become a `Tabs` switcher rather than two columns — reusing the same Radix `Tabs` primitive the Match/Exceptions/History page-local tabs already use, one tier down inside the Match tab itself — and the Match Tray becomes a `sticky bottom-0` bar rather than a center column, carrying the staged count and the Match/Split/Clear actions exactly as the desktop tray does. Tapping a statement line stages it and automatically switches the active tab to Candidates with the plausible counterpart scrolled into view and highlighted, preserving the "pair a line with a candidate" mental model the side-by-side board gives desktop, at the cost of one extra tap rather than a fundamentally different workflow. Because the two-pane board is meaningfully better than the tabbed fallback, opening this screen in tablet portrait shows a one-time, per-device-dismissible hint ("Rotate for the full matching view") using the identical dismissal mechanism named in `# Navigation & Multitasking`'s Stage Manager guidance — a suggestion, never a requirement, since the tabbed fallback is fully functional on its own.

## Dashboards

Both dashboard surfaces this platform documents — the plain `docs/frontend/DASHBOARD.md` home screen and the AI Command Center — treat tablet as the tier where their spatial grid concept first activates, not merely a wider version of mobile's single column. `docs/frontend/DASHBOARD.md`'s own Responsive Behavior table gives the KPI Strip two tiles per row starting at `md`, with the Chart Region/AI Summary Rail 8/4 split and the Aging Summary's side-by-side pairing both waiting for `lg`; `docs/ai/AI_COMMAND_CENTER.md → Layout` states plainly that its own bento grid — collapsed to `mobile_command_feed`'s single swipeable column below the tablet breakpoint — resumes being an actual grid at tablet width. The synthesis this document draws from those two sources: a tablet in **portrait** sees a two-column KPI row with every other dashboard region still stacked full-width beneath it (more structure than mobile's single card-per-swipe, less than the full bento); rotating the same device to **landscape** crosses into `lg` and unlocks the full 8/4 chart-plus-insights split and the paired aging bars, with no additional code path — the identical `xl:col-span-*`/`lg:col-span-*` classes `docs/frontend/RESPONSIVE_DESIGN.md → Dashboard Reflow`'s worked example already declares simply resolve differently as the same element's container crosses the boundary.

## Data entry

`docs/frontend/RESPONSIVE_DESIGN.md → Multi-line editors`'s `JournalLineRow` branches on `isCompact = tier === 'base' || tier === 'sm'` — which means Tablet (`md`) already falls on the **desktop** side of that specific branch and renders the dense, inline, column-based editable row (Account · Debit · Credit · Cost Center · Delete), not Mobile's stacked-field card. This is worth stating explicitly because it is easy to assume tablet needs the mobile treatment when in fact the existing rule already gives it the more efficient one. What tablet does *not* inherit from that same dense row is its row **height** — `docs/frontend/RESPONSIVE_DESIGN.md → Touch Targets & Gestures` sets table rows to 56px on Mobile and Tablet (both), reserving the tighter 40px row for `lg`+ where a mouse offers pixel precision. The concrete, previously-unstated implication this document draws from combining those two existing rules: **a tablet's journal-line row renders every desktop column, at the touch-safe 56px height, not the 40px mouse height** — the same fields, more room around each one, because touch precision still matters on a tablet even once its layout has earned the desktop's column arrangement.

Attachment capture on this same form follows a similar refinement. `docs/frontend/RESPONSIVE_DESIGN.md → Forms On Mobile` gives Mobile a camera-capable file input and "Tablet and up" a drag-and-drop dropzone — a reasonable default for a screen used at a desk, but it underserves the tablet used in the field: a Warehouse Employee photographing a delivery note on an iPad, or an Accountant capturing a paper receipt handed across a counter, needs the camera path just as much as a phone user does. This document specifies that tablet renders **both** affordances together — the dropzone as the primary target (since a tablet paired with a keyboard case and used at a desk, with a Files app window open in Split View, is at least as common a QAYD tablet scenario as a handheld one) and a secondary "Take Photo" button reusing the identical `<input type="file" accept="image/*" capture="environment">` Mobile already uses, both calling the same upload mutation so which affordance a user reaches for never changes what happens to the file afterward. In landscape (`lg`+), the Form Page Template's Live Preview/Help Rail (`docs/frontend/LAYOUT_SYSTEM.md`'s 9/3 or 8/4 split, which "activates for preview rail" specifically at `lg` per that document's per-template table) appears beside the line-item grid — the running debit/credit balance and any AI template suggestion are visible while lines are being entered, not only after; tablet portrait keeps the single-column form with the sticky bottom action bar, identical to Mobile's arrangement, since the preview rail's own minimum width does not fit alongside a usable line-item grid below `lg`.

# Input

## Keyboard

A hardware keyboard — Smart Keyboard Folio, Magic Keyboard, or any Bluetooth keyboard on Android — gives a tablet the platform's full shortcut surface unmodified (`# Navigation & Multitasking`). The one tablet-specific concern this section owns is the **software** keyboard's effect on layout when no hardware keyboard is attached: iPadOS's on-screen keyboard commonly covers 35–40% of an 11" iPad's height in portrait, enough to hide a focused field or a sticky bottom action bar entirely if the layout does not account for it. QAYD's forms already auto-scroll the first invalid field into view on a failed submit (`docs/frontend/RESPONSIVE_DESIGN.md → Sticky submission and error handling`); this document extends the same care to the *focus*, not only the *error*, path — every text/number input's `focus` handler calls `scrollIntoView({ block: 'center' })`, and the sticky bottom action bar reads the `VisualViewport` API's `height`/`offsetTop` (rather than the layout viewport) to reposition itself above the software keyboard instead of being obscured beneath it, the standard, correct mechanism for this exact problem on both iPadOS Safari and Android Chrome. Apple's Scribble (handwriting recognition inside any standard text field) and voice dictation both operate beneath any correctly labeled `<input>`/`<textarea>` QAYD already renders — per `docs/frontend/ACCESSIBILITY.md`'s "Labels, never placeholders" convention — and need no additional implementation to work correctly; a form built to that accessibility standard already gets Scribble for free.

## Pointer

A trackpad or mouse attached to a tablet is treated as a **capability** rather than a **tier**, exactly as `# Navigation & Multitasking` specifies for multi-select: `@media (hover: hover) and (pointer: fine)` unlocks hover-revealed row actions and `Ctrl/Cmd`-click extension regardless of whether the surrounding layout is currently in the Tablet or Laptop tier. `dnd-kit`'s pointer sensor (already named in `docs/frontend/RESPONSIVE_DESIGN.md`'s Gesture catalog for "reordering report widgets, price-list priority… all tiers") gives a tablet with a trackpad the identical precise-drag reordering a laptop mouse gets, alongside — never instead of — the touch sensor every tablet already carries.

## Touch

The platform's 44×44px minimum hit area (`docs/frontend/LAYOUT_SYSTEM.md → Touch targets`, `docs/frontend/RESPONSIVE_DESIGN.md → Touch Targets & Gestures`) applies on tablet without modification, and — stated as an explicit rule rather than left implicit — **a fine pointer being detected never shrinks a touch target back down.** `pointer: fine` is additive (it unlocks hover states and precise click-drag) and must never be read as "the user will not also touch the glass," because a tablet's touchscreen remains live and frequently used even with a keyboard case attached; a user routinely lifts a hand off the trackpad to tap directly. `SwipeableApprovalCard`'s drag-to-decide gesture and the platform's pull-to-refresh and long-press-to-multi-select affordances are all already scoped to "Mobile, Tablet (touch)" in `docs/frontend/RESPONSIVE_DESIGN.md`'s Gesture catalog, so a tablet inherits every one of them unchanged, with the same rule that a gesture is always an accelerator layered on top of an explicit button, never a control's only path (`docs/frontend/RESPONSIVE_DESIGN.md → Swipeable approval card`).

## Stylus — optional, additive, never required

Apple Pencil (and, on Android tablets, an S Pen or equivalent) is in scope for exactly two surfaces this document specifies, both genuinely new to this series and both scoped conservatively:

- **Document annotation during review.** The `AttachmentViewerPane` from `# Priority Tablet Journeys` accepts freehand strokes as an optional overlay — a reviewer circling a line item or writing a short note directly on a vendor bill before rejecting it or requesting changes. Strokes are captured as vector paths with coordinates normalized against the *source document's own* intrinsic bounding box (a ratio of the image's width/height, not the app shell's logical start/end axis — see `# RTL & Localization`), stored alongside the request rather than flattened into the underlying attachment, and rendered to every reviewer in the same approval chain. This is a supplement to, never a replacement for, the mandatory typed `reason`/`note` field `docs/frontend/APPROVAL_CENTER.md`'s Reject and Request Changes `AlertDialog`s already require — an annotation clarifies where on the document a concern sits; it is not itself the auditable record of why a request was rejected.
- **Signature capture on a goods receipt.** A Warehouse Employee or Inventory Manager confirming a `goods_receipts` record against a physical delivery captures the receiving driver's signature via a canvas-based `SignaturePad`, attached through the same polymorphic `attachments` table (`attachable_type: 'goods_receipts'`) every other document in the platform already uses — a consistent application of an existing mechanism, not a new one.

Both surfaces accept **finger or stylus identically** — a canvas capturing `pointerdown`/`pointermove`/`pointerup` events, never a stylus-exclusive API — because not every tablet in the field has a Pencil paired, and neither annotation nor a signature is a required step a business action can be blocked on for want of one. Palm rejection is handled by the browser's own Pointer Events `pointerType` discrimination (`'pen'` versus `'touch'`), not by anything QAYD hand-rolls. Apple Pencil hover/tilt preview (Pencil Pro, 2nd generation) is noted as a `# Future` enhancement to the annotation surface specifically, not a requirement of this version.

# Density & Typography for tablet

Density and text size are governed entirely by the existing **per-user, not per-device** preference `docs/frontend/THEMING.md → Density & Text-Size Preferences` already defines — `user_preferences.density` (`comfortable`/`compact`) and `.text_scale` (`small`/`default`/`large`), applied through the `--density-scale` CSS multiplier and a root `data-text-scale` attribute respectively. This document introduces **no tablet-specific override** of either mechanism, deliberately: a Senior Accountant who prefers Compact density for a 200-line Journal Entries review wants that same density on an iPad as on a laptop, and the platform's own per-table density defaults (Compact for Journal Entries/Ledger/Bank Transactions/Stock Movements; Comfortable for Customers/Vendors/Products/Invoices, per `docs/frontend/LAYOUT_SYSTEM.md → Density Modes`) already produce sensible tablet behavior without modification. What tablet's extra width changes is not density but **column count** — more `priority`-ranked columns become visible at a given density before the platform's column-priority-hiding rule needs to hide any of them (`docs/frontend/LAYOUT_SYSTEM.md → Table responsive strategy`), which is the correct lever, not a device-specific density override that would fragment one user's preference across their own devices.

Typography needs no tablet-specific rule either, and this document states that plainly as an affirmative property of the existing system rather than a gap: `docs/frontend/DESIGN_LANGUAGE.md`'s and `docs/frontend/RESPONSIVE_DESIGN.md`'s fluid type scale is built entirely from `clamp()` expressions anchored between a Mobile (360px) minimum and a Desktop (1280px+) maximum, each combining a `rem` term with a `vw` term specifically so the scale honors both viewport width and the user's own browser/OS text-size preference (`docs/frontend/RESPONSIVE_DESIGN.md → Fluid type scale`). A tablet's 744–1194px width sits, by construction, in the interpolated middle of every one of those curves — a Display heading renders at an intermediate size between the mobile and desktop endpoints with no separate tablet breakpoint required, because that is precisely the problem `clamp()` was chosen to solve. The one place this document does make an explicit recommendation is ergonomic rather than technical: a tablet is used at both a phone's close handheld distance (a stock count in a warehouse aisle) and closer to a monitor's arm's-length distance (propped on a stand behind a keyboard case), and a user who primarily uses their tablet the second way is a good, obvious candidate for the `text_scale: 'large'` preference the platform already ships — this document recommends surfacing that preference prominently in onboarding for a detected tablet form factor, not building a second, parallel type-scaling mechanism to solve the same problem the existing one already solves.

Row-height and touch-target composition follows `docs/frontend/LAYOUT_SYSTEM.md → Density Modes` unchanged: Comfortable's 48px row already clears the 44px touch minimum with margin; Compact's 32px row still requires the documented `::before` hit-slop expansion to reach an effective 44px tap target, and this applies with identical force on a tablet's touchscreen as on a phone's — a keyboard case attached does not make a Compact-density row's tap target any larger, because the finger reaching for it has not gone anywhere.

# Performance

Tablet ships the same JavaScript bundle as every other tier — there is no separate tablet build, code path, or Next.js route group, consistent with `# Purpose`'s founding statement that QAYD is one responsive application. Every performance mechanism already specified elsewhere in this series applies to tablet exactly as written, with the following tier-specific notes this document adds:

- **RSC-first rendering and per-widget Suspense streaming** (`docs/frontend/FRONTEND_ARCHITECTURE.md → Rendering Strategy`) benefit tablet identically to desktop — a slow AI Summary Rail widget never blocks the KPI Strip's first paint on an iPad any more than it does on a laptop, since the streaming boundary is a property of the route, not the viewport.
- **Virtualization thresholds** (`@tanstack/react-virtual`, activating past ~200 rows per `docs/frontend/DESIGN_LANGUAGE.md → Virtualization & interaction`) apply unchanged, with the row-height estimate fed to the virtualizer reading the *actual* active tier-and-density combination — 56px touch rows in Tablet tier, 40px mouse rows once a screen crosses into Laptop tier at `lg` — so a rotation from portrait to landscape mid-session correctly re-estimates rather than leaving stale row-height assumptions behind, exactly the re-`measure()` discipline `docs/frontend/LAYOUT_SYSTEM.md → Density Modes` already requires for a live density switch.
- **Canvas rendering at device pixel ratio.** The `SignaturePad` and any canvas-based sparkline or chart must set the canvas element's backing-store `width`/`height` attributes to `cssWidth × devicePixelRatio` (with a matching CSS `width`/`height` and a `ctx.scale(dpr, dpr)` call) rather than relying on the CSS-sized default — a signature or a KPI sparkline rendered without this correction reads visibly blurred on a Retina-class iPad's 2x display, a defect specific to canvas (unlike an SVG chart, which scales correctly by default) and worth calling out explicitly since this document is the first to introduce a canvas-drawn surface.
- **Realtime subscription scoping follows the device's power profile, not its screen size.** `docs/frontend/RESPONSIVE_DESIGN.md → Realtime subscription scoping` gates a dashboard widget's Echo subscription behind an `IntersectionObserver` on Mobile specifically, for battery and radio reasons, while leaving Desktop's subscriptions unconditional since a plugged-in monitor has no comparable battery concern. Tablet's layout at `lg`+ visually resembles Desktop's bento grid, but its *power profile* resembles Mobile's — a battery-powered, frequently cellular-connected device — so this document specifies that the `IntersectionObserver` gating stays active on tablet at every tier, including Laptop-tier landscape, even though the surrounding grid otherwise looks identical to desktop's ungated version. This is a deliberate case where two of the platform's own responsive axes (layout tier and device power class) diverge, and the correct behavior tracks power class, not layout tier.
- **Offline-tolerant drafts** (debounced IndexedDB autosave keyed by a client-generated `Idempotency-Key`, per `docs/frontend/RESPONSIVE_DESIGN.md → Forms On Mobile`) apply unchanged to tablet, and are specifically relevant to the stock-count and goods-receipt-signature journeys named in `# Priority Tablet Journeys` and `# Input`, both plausibly performed on a warehouse floor with unreliable WiFi.

# Design Adaptation

Nothing in this document introduces a new color, radius, shadow, or motion token, and that is a deliberate constraint rather than an oversight — `docs/frontend/DESIGN_LANGUAGE.md`'s near-monochrome ink scale, single disciplined brass accent, ≤10px corner-radius ceiling, and hairline-border-over-shadow elevation model are stated as governing "every screen built under `Module: Frontend`," with no carve-out for a device tier, and this document does not request one. Tablet's extra canvas, relative to mobile, is spent on **structure**, never decoration: a second pane (master-detail, the document-beside-decision review layout), a second column (the two-tile KPI row), or a preview rail that mobile had no room for (`docs/frontend/LAYOUT_SYSTEM.md`'s Live Preview/Help Rail) — never a denser cluster of colored badges, a bigger illustration, or a rounder card, all of which `docs/frontend/DESIGN_LANGUAGE.md → Design Principle 4` ("density with air") and `→ Design Principle 8` ("consistency compounds; novelty costs") already rule out platform-wide. The temptation a wider canvas creates — filling it with a third or fourth simultaneous widget "because there's room" — is treated in this document exactly as `docs/frontend/LAYOUT_SYSTEM.md → Nesting rule` treats a third level of card nesting: a signal to reconsider the layout, not license to add another surface.

`surface-glass` — the frosted, blurred treatment `docs/frontend/DESIGN_LANGUAGE.md → Elevation & Surfaces` reserves specifically for the Command Palette and the AI Command Center overlay, precisely because its rarity is what makes it read as premium — earns no new tablet exception. A tablet's off-canvas navigation `Sheet` (`# Layout`) and its master-detail split both use the platform's ordinary `surface-1`/`surface-2` treatment, not glass, even though a "glassy sidebar drawer" is a common enough pattern on tablet-class software elsewhere that it might seem like an obvious fit; QAYD's own taste brief is explicit that novelty in this direction costs more than it earns. The one visual element this document notes renders correctly on tablet with zero special-casing, worth stating as a positive result of the one-shell approach rather than leaving it unremarked: the app shell Topbar's `backdrop-filter: blur(8px)` frosted treatment (`docs/frontend/LAYOUT_SYSTEM.md`'s shell CSS) is the same component instance on every tier, and needs no tablet-specific adjustment to look correct at this width.

# RTL & Localization

Every RTL rule this platform's documentation establishes — logical properties exclusively, ESLint-enforced (`docs/frontend/LAYOUT_SYSTEM.md → RTL Layout Mirroring`); numerals, currency codes, and dates staying LTR-embedded via `dir="ltr"`/`unicode-bidi: isolate` spans (`docs/frontend/LAYOUT_SYSTEM.md`'s `Bidi` component); charts never mirroring their internal axis (`Rule 5`); debit/credit column order physically fixed regardless of direction (`Rule 2, Exception B`); directional icons flipping while meaning-bearing icons never do (`Rule 4`) — applies to tablet completely unchanged, and this is worth stating explicitly as a genuine payoff of the logical-properties architecture: **no rule in this document required a tablet-specific RTL carve-out**, because none of the tier-specific decisions above (the container-query split threshold, the 320px list pane, the tabbed reconciliation fallback) reference a physical direction anywhere in their own implementation.

Two new surfaces this document introduces need their own explicit statement, since neither existed in the corpus before:

- **The master-detail split's pane order is logical, not physical**, using the identical mechanism `docs/frontend/LAYOUT_SYSTEM.md → App Shell Layout` already relies on for the Sidebar (`grid-template-columns` with no explicit `direction: ltr` override, so "column 1" is the logical start edge and flips physical side automatically under `dir="rtl"`). The list pane renders at the logical start edge and the detail pane at the logical end edge in both directions — a right-to-left Arabic session sees the list on the physical right and the detail pane on the physical left, mirroring the platform's Sidebar-on-the-opposite-edge behavior rather than introducing a new, list-pane-specific convention.
- **The source document inside `AttachmentViewerPane` never mirrors, regardless of the app's active locale** — a direct extension of Rule 5's underlying principle (chart internals reflect the *data's* own intrinsic order, not the app chrome's direction) applied to a new content type: a vendor's invoice, in whatever language and layout the vendor itself produced it in, renders exactly as scanned. Only the chrome around it (the pane's own header, the zoom controls) mirrors. Annotation strokes captured on top of that document are stored as coordinates normalized against the *image's own* bounding box — an x/y ratio, not a logical-start/logical-end offset — because a stroke is drawn on the document's coordinate space, which has no relationship to the surrounding app's reading direction.

# Accessibility

Tablet targets the identical **WCAG 2.2 AA** conformance floor `docs/frontend/ACCESSIBILITY.md → Standards & Targets` sets platform-wide, verified in both languages and both themes exactly as every other tier requires, with the following tablet-specific applications:

- **Touch targets never shrink because a fine pointer was detected.** Restated from `# Input` because it is an accessibility guarantee, not merely an ergonomic one: `pointer: fine` unlocks additional affordances (hover states, precise click-drag, `Ctrl`-click multi-select) and must never be read as permission to reduce a control below the platform's 44×44px minimum, because a tablet's touchscreen remains a live, simultaneously-available input path even with a keyboard case attached, and a user who relies on touch for motor-accessibility reasons is not accommodated by a control that assumes a trackpad is now the only input in play.
- **Screen readers work against the same DOM and ARIA tree, with no tablet-specific tree.** VoiceOver on iPadOS and TalkBack on Android tablets read the identical landmark structure, heading hierarchy, and live-region behavior (`polite`, never `assertive`, for a realtime dashboard tick or an approvals-queue update, per `docs/frontend/ACCESSIBILITY.md → Live regions`) that desktop and mobile already specify — this document introduces no bespoke accessibility tree for tablet. VoiceOver's rotor navigation and any external-keyboard VoiceOver command (using a paired Bluetooth keyboard's VO modifier) both operate against the same tab order and ARIA roles already defined for every other tier.
- **Split View and Stage Manager reflow must carry the accessibility tree with it, not only the visual layout.** When a tablet's live viewport narrows under Split View or a resized Stage Manager window and a screen reflows from its two-pane form to its single-pane fallback (the master-detail split collapsing to list-then-navigate, the reconciliation board collapsing to tabs), the heading order and landmark order in the accessibility tree must reflow identically — a screen-reader user's heading list should never describe a two-pane layout that the visual layout has already collapsed out from under them. This document recommends extending `docs/frontend/ACCESSIBILITY.md → Testing`'s viewport matrix with at least one genuinely odd width drawn from a real iPad Split View half (390pt, 507pt, 590pt) specifically because these fall between the platform's five named tiers and are exactly the widths a named-tier-only test suite would never otherwise exercise.
- **Reduced motion applies to every new tablet-specific gesture exactly as it applies to the existing ones.** The document-annotation stroke layer and any drag-to-pair enhancement to the reconciliation board (`# Future`) must both honor `useReducedMotion()` by disabling the underlying drag/draw interaction outright — not merely skipping its animation — mirroring `docs/frontend/RESPONSIVE_DESIGN.md → Swipeable approval card`'s own precedent that a user who has asked the OS for reduced motion is frequently asking because the moving gesture itself is the discomfort, and the explicit button/tap alternative must remain the fully sufficient path regardless.

# Edge Cases

| Edge case | Behavior |
|---|---|
| iPad Pro 12.9" in portrait (1024px CSS width) sits exactly on the `lg` boundary | Renders full Laptop-tier chrome — persistent collapsed rail, three-pane reconciliation board — even in portrait, unlike every smaller iPad. This is the correct, intended consequence of CSS-tier-driven design (`# Platform Strategy`), not a special case requiring its own code path; no device-specific override should ever be added to "fix" this into behaving like a smaller iPad's portrait mode. |
| A QAYD window is placed into a narrow iPadOS Split View or Slide Over | The live container width may drop into Mobile tier despite tablet hardware; every screen renders its Mobile-tier layout correctly, because layout is decided by live width, never by device detection (`# Navigation & Multitasking`). No error, no broken layout, no "please use full screen" wall. |
| A Stage Manager window is resized mid-session while a Bank Reconciliation match is staged, or a document annotation is mid-stroke | Client-only state (`useMatchStaging`'s Zustand store, an in-progress annotation draft) is not scoped to a specific breakpoint's component tree and survives the reflow intact — a resize is not a navigation or a remount, so nothing needs to be specially preserved; this already works correctly by virtue of `docs/frontend/FRONTEND_ARCHITECTURE.md`'s state-management architecture. |
| An external keyboard/trackpad is detached mid-session while a `Ctrl`-click multi-selection is active | `useFinePointer()`'s underlying `matchMedia` listener re-evaluates immediately; the moment `pointer: fine` no longer matches, the UI falls back to the always-present checkbox/long-press selection affordance rather than leaving a now-unreachable interaction pattern active with no visible alternative. |
| A goods-receipt signature or a document annotation is captured while offline (a warehouse with poor WiFi) | Queues locally via the same IndexedDB-draft-plus-`Idempotency-Key` mechanism `docs/frontend/RESPONSIVE_DESIGN.md → Forms On Mobile` already specifies for interruption-prone mobile forms, and syncs once connectivity returns — never silently dropped, and never double-submitted on retry. |
| A company-managed (MDM) enterprise iPad has Face ID/Touch ID disabled by policy, or the base model lacks biometrics entirely | The sensitive-action re-confirmation step (`# Feature Availability`) falls back to a device passcode or the company's own SSO step-up re-authentication — it is never silently skipped just because a biometric sensor is unavailable, since the point of the step is an extra confirmation immediately before a bank transfer or payroll release, not specifically a biometric one. |
| The same user reviews the same Approval Center queue from two different tablets (a personal iPad at home, an office-issued Android tablet at a desk) | Each is simply another authenticated browser session against the same company-scoped Reverb channel (`private-company.{id}.approvals`) — realtime invalidation keeps both in sync exactly as any two-tab scenario already does (`docs/frontend/DASHBOARD.md → Edge Cases`'s two-tab branch-filter example extends here without modification), since sessions and Reverb channels are never device-scoped. |
| An Arabic-locale session is opened inside a narrow Split View pane | The logical-properties architecture requires no RTL-specific narrow-width handling, but this document flags the combination of RTL **and** an unusually narrow width as a specific pairing worth adding to the visual-regression matrix (`# Accessibility`) — the sticky reconciliation table's scroll-hint gradient (`bg-gradient-to-l … rtl:bg-gradient-to-r`) and the Topbar's `backdrop-filter` are the two elements most likely to reveal a subtle regression at a width the platform's five named tiers do not individually test. |
| A tablet's software keyboard is dismissed and re-summoned rapidly (switching between two form fields near the top and bottom of a long journal entry) | The `VisualViewport`-driven sticky-bar repositioning (`# Input`) debounces its own reflow rather than recalculating on every intermediate resize event the keyboard's animation fires, so the action bar does not visibly jitter during the keyboard's own show/hide transition. |

# Future

- **A true native tablet shell.** If the platform's distribution strategy ever requires App Store/Play Store presence beyond an installed PWA — a plausible future step given `docs/frontend/RESPONSIVE_DESIGN.md`'s own "native shell / PWA" phrasing already anticipates *some* native wrapper — that shell gets its own specification superseding this document's PWA-scoped guidance; this document does not pre-design it.
- **True multi-window.** iPadOS's genuine multi-window support (opening two independent QAYD windows side by side — Journal Entries in one, the Chart of Accounts as a live reference in the other) is not yet specified. Given how often this platform's own worked examples involve cross-referencing a source document against a ledger entry, this is a plausible, high-value extension once the underlying multi-window web platform APIs (and Safari's own support for them) mature enough to specify concretely.
- **Apple Pencil hover and pressure.** Pencil Pro/2nd-generation hover preview over the `AttachmentViewerPane`'s annotation layer, and pressure-sensitive stroke weight, are natural refinements of the annotation surface `# Input` specifies conservatively today; neither is required for the surface to function, and both are deferred until the base annotation feature has real usage to learn from.
- **Handoff and Universal Clipboard continuity** between an iPad and a Mac or iPhone signed into the same account is unaddressed; a CFO starting a review on an iPad and finishing it on a laptop is a real, named persona-consistent scenario across this documentation series, but the continuity mechanism itself has no specification yet.
- **A more IDE-like pane-focus keyboard scheme**, extending `Cmd/Ctrl+[`/`Cmd/Ctrl+]` (`# Navigation & Multitasking`) toward a fuller `Cmd+1`/`Cmd+2`-style direct pane-focus set, once user research on actual keyboard-cased tablet usage validates demand for it rather than speculatively building it now.
- **External-display and AirPlay presentation mode** for the Dashboard or a Financial Statement, relevant to the Owner/CFO board-review persona this documentation series already establishes, is a plausible extension with no current specification — a dedicated "presentation" density/theme variant (larger type, higher contrast, no interactive chrome) would need its own design pass rather than reusing an existing density preference verbatim.
